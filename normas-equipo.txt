---------------------------------------------------------------------------
¿CÓMO LABURAMOS? (REGLAS DEL GRUPO)
---------------------------------------------------------------------------

- NOMBRES EN ESPAÑOL: Como somos de acá y el proyecto es para la UPE, 
  que les parece cualquier tipo de declaracion que sea en castellano?. Así es 
  más fácil leer el código y nos entendemos todos rápido.

- NO MEZCLAR TODO: Tratemos de que cada cosa esté en su carpeta. No dejar archivos 
  sueltos en la carpeta raíz porque se hace un lío.

- ARCHIVOS BINARIOS: Acuérdense que el profe pidió persistencia binaria. 
  No usar .txt para guardar los datos de las rutas; usamos los .dat que
  están en la carpeta /datos.

- MENSAJES DE COMMITS: Tratemos de que los mensajes de los 'commits' se entiendan.
  No poner solo "cambios", pongamos algo como "Agregada función de cargar ciudad"
  o "Corregido error de objeto ciudad".

- ANTES DE SUBIR AL GITHUB: Porfa, antes de mandar un 'push', asegurarse 
  de que el programa les compila en su compu. Para no subir algo roto.

- NOMBRES DE VARIABLES: Se usará CamelCase (nombreDeLaFuncion).

- INDENTACIÓN: Se optará por la indentación estilo:
	if(condicion)
	{
	}

- FEATURES: A la hora de empezar la implementación de cualquier nueva feature
  o función, se creará una branch de main donde se probará la funcionalidad
  de dicha feature. Cuando se considere lista para agregar, se abrirá una
  Pull Request y el equipo la revisará. Si está todo OK, se mergea la feature
  a la rama main. Esto es para mantener la rama main limpia y funcionando,
  y que un error en el código de alguien no afecte a los demás.
